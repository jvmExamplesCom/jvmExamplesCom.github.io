android.support.annotation.NonNull

The 

          
           
android.support.annotation.NonNull
annotation is a built-in Java annotation from the Android Support Annotations library which indicates that a given parameter must not be null. It is used to ensure that code does not accept null values as parameters, and can help improve the robustness of an application.

Introduction to android.support.annotation.NonNull

The 

          
           
android.support.annotation.NonNull
annotation is a type of structured annotation which exists in the Android framework. This annotation is used to denote parameters which must not be null. The annotation is designed to document which parameters must not have a null value, and this helps communicate the expectations of the programmer more clearly. It is important to note that the 

          
           
android.support.annotation.NonNull
annotation is not enforced by the compiler, and it does not change the behavior of the code. It is only a marker that communicates to other developers that the code expects a non-null value for the parameter.

Using android.support.annotation.NonNull

The 

          
           
android.support.annotation.NonNull
annotation can be used to mark the parameters which must not be null. To do so, simply annotate the parameter with 

          
           
@NonNull
: 

          
           
public void foo(@NonNull String param) {
    // code goes here
}
It is possible to use this annotation on both method parameters and local variables. This helps document which variables must not be null and can help improve the robustness of the codebase.

Benefits of Using android.support.annotation.NonNull

Using the 

          
           
android.support.annotation.NonNull
annotation can help improve the robustness of a codebase by communicating which parameters must not be null. By annotating the parameters which must not have a null value, it helps make the code more readable and easier to understand. Additionally, the annotation can help detect potential errors early on in the development process. If a parameter which must not have a null value is passed a null value, the application may crash or behave unexpectedly.
