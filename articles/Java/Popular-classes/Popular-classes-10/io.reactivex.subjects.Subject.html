ReactiveX is an API that allows developers to create reactive programs using observable sequences. This article will focus on the Subject type of ReactiveX, specifically the io.reactivex.subjects.Subject that is part of the io.reactivex.rxjava package.

What is a Subject?

A Subject is a special type of Observable that is both an Observable and an Observer at the same time. This makes Subjects particularly useful for multicasting (sending a single emission to multiple Observers). As an Observable it can emit events, and as an Observer it can receive them.

Creating a Subject

There are several different types of Subjects available in the io.reactivex.subjects package, including PublishSubject, ReplaySubject, and UnicastSubject. To create an instance of one of these Subjects, you can simply use their constructors. For example: 

          
           
PublishSubject<String> subject = PublishSubject.create();
This creates a PublishSubject object that is subscribed to type String. The other Subjects would require similar creation statements.

Using a Subject

Subjects are used to multicast emissions to multiple Observers. To do this, they must first be subscribed to by an Observer. This is done using the subscribe() method. 

          
           
subject.subscribe(stringObserver);
This subscribes the stringObserver to the Subject. Once subscribed, the Subject can emit items to the Observer. This is done using the onNext() method. 

          
           
subject.onNext("Hello");
This will send the emission “Hello” to the stringObserver. Any other Observers that have previously subscribed to the Subject will also receive this emission. The Subject can emit any type of emission — not just Strings. It can send any type of Object, including custom objects, or primitive values such as booleans or integers.

Conclusion

Subjects are a powerful tool for multicasting emissions to multiple Observers. They are easy to create and use, and can be used to send any type of Object. They are an important part of the ReactiveX API and should be understood by any serious ReactiveX developer.
