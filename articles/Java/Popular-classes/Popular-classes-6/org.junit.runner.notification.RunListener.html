org.junit.runner.notification.RunListener

The RunListener class is an essential component of the JUnit testing framework. RunListeners are responsible for listening to events related to the running of tests and reacting accordingly. They provide a clean way to isolate functionality related to running tests from the implementation of tests themselves.

Overview

The org.junit.runner.notification.RunListener class is an abstract class that defines the basic interface for monitoring the running of tests. As a first step to using a RunListener, you must create an instance of one of its subclasses. A RunListener subclass can then be passed to an instance of org.junit.runner.Runner. Once this is done, the Runner will then invoke the corresponding RunListener methods when it performs various actions such as starting and finishing a test or throwing an exception.

Example Implementation

Below is an example of a RunListener subclass which prints out messages to the console when tests start and complete. 

          
           
public class LogRunListener extends RunListener {

    public void testStarted(Description description) {
        System.out.println("Test started: " + description);
    }

    public void testFinished(Description description) {
        System.out.println("Test finished: " + description);
    }
}
This RunListener subclass can then be used as follows: 

          
           
LogRunListener listener = new LogRunListener();
Runner runner = Request.aClass(MyTestClass).getRunner();
runner.addListener(listener);
Result result = runner.run();
When the Runner instance is created, it will invoke the corresponding methods of the RunListener instance. In the case of the example above, the output would be similar to this: 

          
           
Test started: [MyTestClass]
Test finished: [MyTestClass]

Conclusion

The RunListener class provides an easy way to listen to events related to the running of tests and perform custom code execution in response. RunListener subclasses can be easily created to implement custom logic and these can then be used to provide more detailed information on the running of tests.
